## Файловая система и права

### Осмотр дисков и монтирование

```bash
lsblk -f                        # блочные устройства, ФС, метки
df -hT                          # использование диска с типами ФС
mount                           # список смонтированных ФС
findmnt                         # дерево монтирования
blkid                           # UUID и типы ФС

mount /dev/sdb1 /mnt/data       # смонтировать раздел
umount /mnt/data                # отмонтировать
mount -o remount,rw /          # перемонтировать / в режим rw
```

> ⚡ Tip: Для работы с дисками и LVM удобно отдельной заметкой держать шаблоны — можно завести `[[linux-cheatsheet]]` и расширить его.

### Базовые права (rwx)

```bash
ls -l                           # rwx-права для user/group/other
ls -ld /var/log                 # права на каталог

chmod 644 file                  # rw-r--r--
chmod 600 id_rsa                # rw-------
chmod 755 script.sh             # rwxr-xr-x (часто для скриптов)
chmod -R 750 /srv/app           # рекурсивно задать права

umask                           # текущая маска по умолчанию
umask 027                       # новые файлы: 640, каталоги: 750
```

Таблица прав:

| Бит | Значение | Для файла         | Для каталога                      |
|-----|----------|-------------------|-----------------------------------|
| r   | read     | чтение            | просмотр списка файлов            |
| w   | write    | запись            | создание/удаление/переименование |
| x   | execute  | выполнение        | вход в каталог (`cd`)            |

### chmod в числах и символах

```bash
chmod 640 file                  # u=rw, g=r, o=-
chmod u+x script.sh             # добавить x только пользователю
chmod g-w file                  # убрать w у группы
chmod o-rwx file                # убрать все права у остальных
chmod a+r file                  # всем добавить r
```

### Владелец и группа (chown/chgrp)

```bash
chown user file                 # сменить владельца
chown user:group file           # сменить владельца и группу
chgrp devops file               # сменить группу
chown -R appuser:app /srv/app   # рекурсивно сменить владельца
```

> ⚠️ Warning: `chown -R /` или по крупным каталогам (`/var`, `/etc`) может серьёзно повредить систему. Меняйте владельца только точечно.

### Спецбиты: setuid, setgid, sticky

```bash
chmod u+s /usr/bin/somebin      # setuid: запуск с правами владельца
chmod g+s /srv/shared           # setgid: новые файлы получают группу каталога
chmod +t /tmp                   # sticky: удалять может только владелец файла
```

Таблица спецбитов:

| Бит    | Обозначение | Пример       | Значение                                      |
|--------|-------------|--------------|-----------------------------------------------|
| setuid | s в user    | `-rwsr-xr-x` | процесс запускается с uid владельца          |
| setgid | s в group   | `-rwxr-sr-x` | процесс/файлы получают gid группы            |
| sticky | t в other   | `drwxrwxrwt` | в каталоге файл может удалить только владелец|

### ACL (Access Control List)

```bash
getfacl file                     # посмотреть ACL
setfacl -m u:deploy:rwx file     # дать пользователю deploy rwx
setfacl -m g:devops:rx file      # группе devops rx
setfacl -x u:deploy file         # удалить запись для deploy
setfacl -b file                  # убрать все ACL
setfacl -R -m u:deploy:rwx dir   # рекурсивно

# ACL по умолчанию для каталога
setfacl -m d:u:deploy:rwx dir    # default ACL для новых файлов
```

### Ссылки (hard/soft)

```bash
ln file1 file2                  # hard link (тот же inode)
ln -s /opt/app/current app      # символическая ссылка
ls -li                          # показывать inode
readlink -f app                 # полный путь цели ссылки
```

### Поиск по правам и владельцам

```bash
find /var/www -type f -user www-data
find /var/www -type f -group www-data
find /var/www -type f -perm 600
find /var/www -type f -perm -o+w       # world-writable
find / -xdev -type f -perm -4000       # все setuid-файлы (без перехода на другие FS)
```

### Журналирование и временные файлы

```bash
ls -lah /var/log                       # системные логи
journalctl -u nginx                    # логи сервиса nginx (см. [[processes-and-services]])

ls -lah /tmp                           # временные файлы
tmpreaper /tmp                         # чистка по политике (если установлен)
```

### Работа с большими деревьями директорий

```bash
du -sh /var/* | sort -h               # кто съел место
du -sh .[!.]* * | sort -h            # включая «скрытые» каталоги

find /var/log -type f -name "*.log" -mtime +7 -delete  # ротация старых логов
```

### Файловые дескрипторы и лимиты

```bash
ulimit -n                              # текущий лимит открытых файлов
ulimit -n 65535                        # временно поднять лимит для текущей shell

cat /etc/security/limits.conf          # постоянные лимиты пользователей
sysctl fs.file-max                     # системный лимит
```

### Таблица chmod (часто используемые значения)

| Значение | Права         | Типичное использование           |
|----------|---------------|----------------------------------|
| 600      | rw-------     | приватные ключи, секреты        |
| 640      | rw-r-----     | конфиги, читаемые группой       |
| 644      | rw-r--r--     | обычные файлы                   |
| 750      | rwxr-x---     | каталоги приложений             |
| 755      | rwxr-xr-x     | исполняемые скрипты/бинарники   |
| 770      | rwxrwx---     | общие каталоги для группы       |
| 777      | rwxrwxrwx     | максимально открытый доступ     |

> ⚡ Tip: Избегайте 777 почти везде. Если что‑то требует 777 — обычно это проблема прав владельца или группы, а не необходимость открывать всё всем.

## Gotchas

- **Случайная смена владельца на / или /var**  
  - **Проблема**: `chown -R app /var` может сломать systemd, логи и кучу системных сервисов.  
  - **Решение**: используйте конкретные пути (`/var/www/app`), а перед выполнением показывайте команду через `echo`.

- **Неправильный umask для сервисов**  
  - **Проблема**: сервис пишет файлы, недоступные другим процессам (например, логам не может прочитать fluentd).  
  - **Решение**: явно задавайте `UMask=` в unit‑файлах systemd (см. `[[processes-and-services]]`) и проверяйте `umask` в shell.

- **ACL перекрывают chmod**  
  - **Проблема**: вы меняете `chmod`, но доступ не меняется.  
  - **Решение**: проверять ACL через `getfacl`, если есть — либо править их, либо очищать `setfacl -b`.

- **Sticky bit на общих каталогах**  
  - **Проблема**: без sticky (`+t`) один пользователь может удалять файлы другого в общем каталоге.  
  - **Решение**: на общих каталогах (`/tmp`, шареные upload‑директории) всегда включать sticky: `chmod 1777 /path`.

> ⚠️ Warning: Массовые операции `chmod -R`/`chown -R` по корню проекта с бинарями, контейнерными томами или системными каталогами могут приводить к трудноотлавливаемым багам и падениям. Сначала запускайте `find`/`ls` для проверки затрагиваемых путей.
