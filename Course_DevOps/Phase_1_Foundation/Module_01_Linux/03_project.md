## Module 01 — Mini‑project: Bash Monitoring Script

### 1. Цель проекта

**Задача**: написать надёжный bash‑скрипт, который:

- периодически проверяет состояние ключевых ресурсов сервера:
  - загрузку CPU;
  - использование RAM;
  - свободное место на основных файловых системах;
  - статус критичных сервисов (например, `nginx`, `docker`, `sshd`);
- пишет результаты в лог‑файл в структурированном виде;
- может работать как:
  - однократный запуск вручную;
  - системный сервис (systemd) с периодическим запуском через timer.

> [!info] Реальный контекст
> Такой скрипт часто используют:
> - на тестовых серверах без полноценного мониторинга;
> - как временное решение до внедрения Prometheus/Grafana (`[[DevOps/08_Monitoring/prometheus]]`, `[[DevOps/08_Monitoring/grafana]]`);
> - как вспомогательный инструмент при разборе инцидентов (`[[DevOps/12_Soft-Skills-and-Processes/incident-response]]`).

Оценочное время: **3–6 часов** (в зависимости от опыта с bash).

---

### 2. Требования к функционалу

Скрипт `server-monitor.sh` должен:

1. Принимать параметры:
   - `-i <interval>` — интервал между измерениями в секундах (по умолчанию 60);
   - `-c <count>` — сколько раз сделать измерение (по умолчанию бесконечно до Ctrl+C);
   - `-s <services>` — список сервисов для проверки, через запятую (например, `nginx,docker,sshd`).
2. Логировать в файл:
   - путь по умолчанию: `/var/log/server-monitor.log` (можно сделать конфигурируемым через переменную окружения `MONITOR_LOG`).
3. На каждой итерации писать:
   - timestamp в ISO‑формате;
   - текущую загрузку CPU (например, среднее по 1 минуте или `top`/`uptime`);
   - использование памяти (`free -h`/`free` в процентах);
   - свободное место по основным файловым системам (`df -h / /var /home`);
   - статус указанных сервисов (`systemctl is-active`).
4. Корректно обрабатывать:
   - Ctrl+C (завершается аккуратно, пишет строку о завершении);
   - невозможность записать в лог‑файл (сообщает в stderr и выходит с ненулевым кодом).

### 3. Технические требования к реализации

- Bash‑скрипт:

```bash
#!/usr/bin/env bash
set -euo pipefail
IFS=$'\n\t'
```

- Использовать:
  - функции с `local` переменными;
  - `getopts` для парсинга аргументов;
  - `trap` для обработки `EXIT`, `INT`, `TERM`;
  - безопасные конструкции с кавычками `"${VAR}"`.
- Вывод:
  - в лог — структурированный текст (подойдёт простой формат, например, «ключ=значение» по строкам);
  - в stdout — минимальная информация (например, «итерация N, всё ок»).

Рекомендуется опираться на паттерны из `[[DevOps/01_Linux/shell-scripting]]` и `[[DevOps/01_Linux/processes-and-services]]`.

### 4. Пошаговый план реализации

#### Шаг 1 — Каркас скрипта

- Создайте директорию `~/projects/server-monitor`.
- Создайте файл `server-monitor.sh` со стандартным каркасом:
  - `set -euo pipefail`;
  - `log()` функция;
  - `main()` и `main "$@"`.
- Добавьте переменные по умолчанию:
  - `INTERVAL=60`;
  - `COUNT=0` (0 = бесконечно);
  - `SERVICES=("nginx" "docker" "sshd")` (по умолчанию).

#### Шаг 2 — Парсинг аргументов

- Реализуйте `getopts`:
  - `-i` для интервала;
  - `-c` для количества итераций;
  - `-s` для списка сервисов (строка вида `nginx,docker` → массив).
- Добавьте функцию `usage()` и вызов при неверных аргументах.

#### Шаг 3 — Инициализация лог‑файла

- Переменная:

```bash
LOG_FILE="${MONITOR_LOG:-/var/log/server-monitor.log}"
```

- Перед началом:
  - создайте родительский каталог, если его нет;
  - проверьте возможность записи (например, `touch "$LOG_FILE"`).
- Настройте перенаправление:

```bash
exec > >(tee -a "$LOG_FILE") 2>&1
```

или вариант, где часть информации идёт только в лог, часть — в stdout (на ваш выбор).

#### Шаг 4 — Сбор метрик за одну итерацию

Создайте функцию `collect_metrics_once()`, которая:

1. Получает timestamp:

```bash
local now
now="$(date -Is)"
```

2. Собирает данные:
   - загрузку CPU (можно через `uptime` или `grep 'cpu ' /proc/stat` с расчётом, на старте допустим упрощённый вариант);
   - использование памяти (`free -m` или `free -h`);
   - свободное место (`df -h / /var /home`);
   - статусы сервисов (`systemctl is-active "$svc"`).
3. Пишет блок в лог:

```text
=== 2026-03-06T12:00:00+00:00 ===
cpu_load=...
mem_used=...
disk_root=...
service_nginx=active
service_docker=active
...
```

#### Шаг 5 — Цикл мониторинга

- В `main()` реализуйте цикл:
  - если `COUNT == 0` → бесконечный while;
  - иначе — for/while до `COUNT` итераций.
- После каждой итерации:
  - `sleep "$INTERVAL"`, если это не последняя.

#### Шаг 6 — Обработка сигналов и завершения

- Реализуйте `cleanup()`:
  - логирует, что скрипт завершён (с кодом выхода);
  - не удаляет лог, только пишет строку.
- Привяжите `trap cleanup EXIT`.
- Дополнительно:
  - `trap 'log "Interrupted by user (SIGINT)"; exit 130' INT`.

#### Шаг 7 — Обёртка в systemd + timer (опционально, но желательно)

- Создайте `/etc/systemd/system/server-monitor.service`:

```ini
[Unit]
Description=Server Monitoring Script

[Service]
Type=simple
User=root
ExecStart=/usr/bin/env bash /home/USER/projects/server-monitor/server-monitor.sh -i 60 -c 0
Restart=always
RestartSec=10
```

- Создайте `/etc/systemd/system/server-monitor.timer`:

```ini
[Unit]
Description=Run server monitor periodically

[Timer]
OnBootSec=30s
OnUnitActiveSec=60s
Unit=server-monitor.service

[Install]
WantedBy=timers.target
```

- Выполните:

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now server-monitor.timer
sudo systemctl list-timers | grep server-monitor
```

См. паттерны из `[[DevOps/01_Linux/processes-and-services]]`.

### 5. Acceptance criteria (как понять, что всё сделано правильно)

Проект считается завершённым, если:

1. **Скрипт корректен технически**:
   - запускается с шебангом `#!/usr/bin/env bash`;
   - использует `set -euo pipefail` и кавычки вокруг переменных;
   - аккуратно обрабатывает аргументы (`-i`, `-c`, `-s`) и ошибки (отсутствие прав, несуществующий сервис).
2. **Функциональность**:
   - при запуске `./server-monitor.sh -i 5 -c 3 -s nginx,sshd`:
     - создаётся/дополняется лог‑файл;
     - в логе три блока измерений с метриками и статусами сервисов.
   - при остановке Ctrl+C в логе появляется запись о прерывании.
3. **Сервисный режим**:
   - unit‑файл `server-monitor.service` валиден и запускается;
   - `systemctl status server-monitor` показывает активный сервис;
   - в `/var/log/server-monitor.log` регулярно появляются новые записи.
4. **Качество кода**:
   - нет «магических чисел» и дублирования (интервалы и пути вынесены в переменные);
   - код структурирован по функциям;
   - присутствуют короткие комментарии только там, где логика неочевидна.

### 6. Stretch goals (для углубления)

Если базовый проект уже работает, можно усложнить:

1. **JSON‑формат логов**  
   Писать каждую строку как JSON‑объект:

```json
{"ts":"2026-03-06T12:00:00Z","cpu_load":0.12,"mem_used_pct":43.5,"service_nginx":"active"}
```

Это упростит последующий парсинг (например, в Loki/ELK — `[[DevOps/08_Monitoring/loki-and-logging]]`, `[[DevOps/08_Monitoring/elk-stack]]`).

2. **Отправка метрик в Prometheus pushgateway**  
   Если у вас есть стенд с Prometheus, попробуйте отправлять метрики через pushgateway и строить графики (`[[DevOps/08_Monitoring/prometheus]]`).

3. **Конфигурационный файл**  
   Вынести список сервисов, интервал и путь до лога в конфиг‑файл (например, INI/YAML) и читать его из скрипта.

4. **Алерт по email/Telegram**  
   Если какой‑то сервис упал или диск свободен <20%, отправлять уведомление (простым хуком или через имеющийся бот/скрипт).

### 7. Связь с дальнейшими фазами

- В модулях по Docker/Kubernetes вы будете переводить подобную логику в:
  - healthcheck'и контейнеров (`[[DevOps/03_Docker/dockerfile-best-practices]]`, `HEALTHCHECK`);
  - Kubernetes liveness/readiness probes (`[[DevOps/04_Kubernetes/workloads]]`);
  - полноценный мониторинг на базе Prometheus/Grafana.
- В модуле по Ansible этот скрипт можно превратить в роль, которая:
  - деплоит скрипт на сервера;
  - настраивает systemd‑сервис и timer (`[[DevOps/07_Ansible/playbooks-and-roles]]`).

Если вы довели проект до рабочего состояния со stretch‑задачами — ваш Linux‑уровень уже ближе к Junior+/Middle в части практических навыков администрирования.
