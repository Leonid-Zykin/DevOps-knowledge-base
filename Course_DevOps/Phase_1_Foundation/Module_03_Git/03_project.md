## Module 03 — Mini‑project: Team Git Workflow Simulation

### 1. Цель проекта

Смоделировать работу небольшой команды над DevOps‑репозиторием:

- несколько разработчиков ведут ветки фич;
- есть основная ветка `main` и protection‑правила (мысленно);
- используется понятная стратегия ветвления и сообщений коммитов;
- возникают конфликты, откаты и hotfix.

Проект нацелен на закрепление стратегий из `[[DevOps/02_Git/branching-strategies]]` и приёмов из `[[DevOps/02_Git/troubleshooting]]`.

Оценочное время: **3–6 часов**.

---

### 2. Описание сценария

Вообразите, что у вас есть репозиторий `infra-config` с такой структурой:

```text
infra-config/
  k8s/
    dev/
    prod/
  ci/
    gitlab/
    github/
  terraform/
    network/
    cluster/
```

Реально создавать всю структуру необязательно — достаточно парочки каталогов и файлов. Важно то, как вы будете с ними работать:

- вы играете за **двух разработчиков** (можно делать это на одной машине, меняя имя/почту);
- работаете по **Trunk-based**: основная ветка `main`, короткоживущие feature‑ветки;
- используете **осознанные коммиты и PR‑подход** (даже если реально не создаёте PR на GitHub/GitLab).

---

### 3. Шаги реализации

#### Шаг 1 — Подготовка «удалённого» репозитория

1. Создайте новый репозиторий на GitHub/GitLab или локальный bare‑репозиторий:

```bash
mkdir -p ~/team-git-sim
cd ~/team-git-sim
git init --bare infra-config.git
```

2. Клонируйте его как «разработчик A»:

```bash
cd ~
git clone ~/team-git-sim/infra-config.git infra-config-A
cd infra-config-A
```

3. Настройте user.name/user.email для A.

#### Шаг 2 — Базовая структура и первый релиз

1. Создайте простую структуру:

```text
k8s/dev/app.yaml
k8s/prod/app.yaml
ci/gitlab/.gitlab-ci.yml
terraform/network/main.tf
```

2. Содержимое может быть минимальным (комментарии/заглушки).
3. Сделайте несколько коммитов в `main`:
   - `chore: bootstrap infra-config repo`;
   - `feat: add dev k8s manifests`;
   - `feat: add basic gitlab-ci pipeline`.
4. `git push -u origin main`.

#### Шаг 3 — Работа двух разработчиков над разными фичами

1. От имени A создайте ветку `feature/dev-env-hardening`:

```bash
git switch -c feature/dev-env-hardening
```

2. Внесите изменения, касающиеся `k8s/dev` и `.gitlab-ci.yml` (например, добавьте stage `lint`, namespace `dev`).
3. Сделайте 2–3 коммита с осмысленными сообщениями.

4. Сымитируйте второго разработчика B:
   - либо реальный второй клон из `infra-config.git` в `~/infra-config-B`;
   - либо изменениям A присвойте другой `user.name/email`.

5. Разработчик B создаёт ветку `feature/prod-deploy-strategy` от `main` и меняет `k8s/prod/app.yaml` (например, replicas, ресурсы).
6. B делает 1–2 коммита и пушит ветку.

#### Шаг 4 — Имитация PR/MR и merge

1. Для каждой ветки:
   - посмотрите diff с main: `git diff main..feature/dev-env-hardening`;
   - убедитесь, что коммиты логичны и небольшие.
2. Сымитируйте code review:
   - напишите себе короткий список замечаний в текстовом файле;
   - при необходимости добавьте ещё один «fixup»‑коммит.
3. Смёржите ветки в main:
   - либо через GitHub/GitLab (рекомендуется);
   - либо локально: `git checkout main && git merge --no-ff feature/... && git push`.

#### Шаг 5 — Конфликт и его решение (горячий merge)

1. Создайте ситуацию конфликта:
   - A меняет `ci/gitlab/.gitlab-ci.yml` (например, stage `deploy-prod` → `deploy-production`);
   - B в параллельной ветке меняет тот же участок файла.
2. Попробуйте смёржить ветки в main и получите конфликт.
3. Разрешите конфликт грамотно:
   - выберите финальный вариант имени стадии;
   - убедитесь, что syntax YAML не сломан.

#### Шаг 6 — Ошибочный релиз и откат

1. Представьте, что один из merge в main сломал пайплайн (допишите в `.gitlab-ci.yml` намеренно ошибочную строку, чтобы `lint` не прошёл).
2. Зафиксируйте этот коммит в main.
3. Затем выполните:

```bash
git log --oneline
git revert <hash_сломавшего_коммита>
git push
```

4. Сымитируйте, что после revert пайплайн снова зелёный.

#### Шаг 7 — Исправление «не там» сделанного коммита

1. Смоделируйте ошибку:
   - сделайте commit в main, который должен был уйти в `feature/infra-docs`.
2. Воспользуйтесь приёмом из `[[DevOps/02_Git/troubleshooting]]`:

```bash
git branch feature/infra-docs HEAD    # создать ветку на текущем коммите
git reset --hard HEAD~1              # убрать коммит из main
git push origin main --force-with-lease # только если понимаете последствия (в реальности — осторожно)
```

3. Либо используйте более «мягкий» путь с revert’ом и cherry‑pick.

### 4. Acceptance criteria

Проект считается завершённым, если:

1. **Репозиторий**:
   - существует общий «удалённый» (real remote или bare‑репо);
   - есть основная ветка `main`;
   - есть несколько feature‑веток, история которых понятна по `git log --oneline --graph`.
2. **Workflow**:
   - сделаны минимум две независимые feature‑ветки;
   - вы смоделировали конфликт merge и корректно его разрешили;
   - выполнен хотя бы один безопасный откат через `git revert`;
   - хотя бы один раз использовались `git reflog` и/или `git stash` в полезном сценарии.
3. **Документация**:
   - есть файл `WORKFLOW_NOTES.md` или похожий, где:
     - описана выбранная стратегия ветвления (например, trunk‑based с короткими feature‑ветками) со ссылкой на `[[DevOps/02_Git/branching-strategies]]`;
     - приведён пример naming‑конвенций веток и сообщений коммитов;
     - описаны правила отката (когда `revert`, когда `reset`).

### 5. Stretch goals

1. **Minimal Git hooks пакет для репозитория**  
   - добавьте pre-commit для проверки формата YAML (через `yamllint`/`docker run ...`);
   - добавьте commit‑msg hook, требующий Conventional Commits (см. пример в `[[DevOps/02_Git/hooks-and-automation]]`).

2. **Интеграция с CI**  
   - подключите реальный GitLab CI/GitHub Actions для проверки `terraform validate`/`kubeval`/`yamllint` при каждом PR;
   - задокументируйте, какие ветки и теги триггерят какие пайплайны (`[[DevOps/05_CI-CD/gitlab-ci]]`, `[[DevOps/05_CI-CD/github-actions]]`).

3. **Связь с GitOps**  
   - добавьте в заметку раздел, где опишете, как бы этот репозиторий использовался с Argo CD (`[[DevOps/05_CI-CD/argocd]]`):
     - какая часть — «config‑repo»;
     - какие ветки/каталоги смотрел бы Argo CD.

### 6. Переход к следующей фазе

Смоделировав такой workflow, вы:

- лучше поймёте, как будут жить репозитории инфраструктуры и Helm‑чартов;
- будете готовы к работе с CI/CD (Phase 5) и GitOps;
- избежите множества типичных ошибок с Git в реальных DevOps‑проектах.
