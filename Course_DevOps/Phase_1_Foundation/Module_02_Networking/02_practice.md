## Module 02 — Networking Practice

Минимальная цель — уверенно пользоваться `ip`, `ss`, `curl`, `dig`, `tcpdump` и базовым firewall на хосте. Все задания выполняйте на своей VM/WSL2/тестовом сервере.

Мы активно используем `[[DevOps/01_Linux/networking]]` и `[[DevOps/11_Networking/troubleshooting]]`.

---

### Task 1 — Карта сети вашей машины

- **Goal**: Понять, какие интерфейсы и IP‑адреса есть на хосте.
- **Steps**:
  1. Выполните `ip -brief address` и `ip r`.
  2. Отдельно выпишите:
     - какой интерфейс смотрит «наружу» (через который идёт default‑маршрут);
     - какие приватные сети у вас есть (10.x/192.168.x/172.16–31.x).
- **Expected result**: Текстовый отчёт `~/net-map.txt` с описанием интерфейсов и маршрутов.
- **How to verify**:
  - В отчёте указаны имена интерфейсов, их IP и какая сеть default.

---

### Task 2 — Проверка DNS и доступности

- **Goal**: Научиться отличать проблему DNS от проблемы маршрутизации.
- **Steps**:
  1. Выберите 2–3 доменных имени (например, `example.com`, `google.com`, `github.com`).
  2. Для каждого:
     - `dig +short <имя>`;
     - `ping -c 4 <имя>`;
     - `ping -c 4 <полученный IP>`.
- **Expected result**: Понимание, где ломается цепочка: DNS или IP‑доступ.
- **How to verify**:
  - Сможете привести пример, когда DNS работает, а IP недоступен (или наоборот), и объяснить, что это означает.

---

### Task 3 — Кто слушает мои порты?

- **Goal**: Освоить `ss` и `lsof` для поиска слушающих сокетов.
- **Steps**:
  1. Установите и запустите любой простой HTTP‑сервер (например, `python -m http.server 8000` в отдельном терминале).
  2. В другом терминале выполните:

```bash
ss -tulpn | grep 8000
sudo lsof -i :8000
```

  3. Убедитесь, что видите процесс и порт.
- **Expected result**: Вы понимаете, как найти, какой процесс слушает конкретный порт.
- **How to verify**:
  - Назовёте PID и команду процесса, слушающего `8000`.

---

### Task 4 — Простейший HTTP‑диагноз с curl

- **Goal**: Научиться использовать `curl -v` для диагностики HTTP.
- **Steps**:
  1. Поднимите локальный сервер (как в Task 3) или используйте внешний (`https://example.com`).
  2. Выполните:

```bash
curl -v http://127.0.0.1:8000
curl -I https://example.com
curl -vk https://self-signed.badssl.com
```

  3. Обратите внимание на:
     - статус‑коды;
     - заголовки `Server`, `Location`, `Content-Type`;
     - TLS‑рукопожатие и ошибки сертификаций.
- **Expected result**: Умение «читать» вывод `curl -v`.
- **How to verify**:
  - Сможете по выводу `curl -v` объяснить, что именно не нравится клиенту (например, self‑signed cert).

---

### Task 5 — Базовый iptables‑firewall (на тестовой машине!)

- **Goal**: Потренироваться с простым firewall‑набором.
- **Steps**:
  1. Убедитесь, что вы работаете в VM/WSL/test, не в проде.
  2. Посмотрите текущие правила:

```bash
sudo iptables -L -n -v
```

  3. Добавьте правило, разрешающее входящий SSH (если его нет):

```bash
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT
```

  4. Временно задайте `DROP` по умолчанию для INPUT:

```bash
sudo iptables -P INPUT DROP
```

  5. Из отдельной SSH‑сессии убедитесь, что доступ не потерян.
  6. Верните политику в исходное состояние (например, ACCEPT или то, что было).
- **Expected result**: Вы увидели на практике, как правила влияют на доступ.
- **How to verify**:
  - Может показать вывод `iptables -L` до и после;
  - можете обосновать, почему такие эксперименты нельзя делать бездумно на прод‑хостах.

> [!warning] Делайте это только там, где вы можете легко восстановиться (консоль/VM/локальный гипервизор). В облаках и проде firewall правится с осознанием последствий.

---

### Task 6 — Захват HTTP‑трафика tcpdump’ом

- **Goal**: Увидеть реальные HTTP‑запросы «на проводе».
- **Steps**:
  1. Запустите тестовый HTTP‑сервер (как в Task 3).
  2. В одном окне терминала:

```bash
sudo tcpdump -i any port 8000 -nn
```

  3. В другом окне:

```bash
curl http://127.0.0.1:8000
```

  4. Посмотрите, какие пакеты появляются (SYN, ACK, данные).
- **Expected result**: Базовое понимание, как выглядит TCP‑сессия и HTTP‑запросы в tcpdump.
- **How to verify**:
  - Сможете показать строки SYN/ACK и объяснить, что происходит.

---

### Task 7 — «Break it» с /etc/hosts

- **Goal**: Увидеть влияние локальных записей `/etc/hosts` на DNS.
- **Steps**:
  1. Выберите домен, который вы часто посещаете (например, `example.com`).
  2. Сначала выполните:

```bash
dig example.com
ping -c 2 example.com
```

  3. Временно добавьте в `/etc/hosts` строку:

```text
203.0.113.123 example.com
```

  4. Повторите `dig +short example.com` и `ping example.com`.
  5. Удалите строку из `/etc/hosts` после эксперимента.
- **Expected result**: Понимание, как локальные записи переопределяют DNS.
- **How to verify**:
  - Можете объяснить, почему `ping` пошёл на другой IP, даже если DNS‑записи не менялись.

---

### Task 8 — Диагностика «приложение не отвечает»

- **Goal**: Отработать типичный сценарий: сервис не открывается по HTTP.
- **Steps** (см. `[[DevOps/11_Networking/troubleshooting]]`):
  1. Создайте тестовый сервис: `python -m http.server 8080`.
  2. В отдельном окне попробуйте:
     - `curl http://127.0.0.1:8080`;
     - `curl http://<ваш_IP>:8080` с другой машины (если есть).
  3. Представьте, что `curl` с другой машины падает. Пройдите чек‑лист:
     - проверка `ip r`, `ss -tulpn | grep 8080`;
     - проверка firewall/iptables/облачных SG;
     - проверка логов приложения (если есть).
- **Expected result**: У вас есть пошаговый план диагностики.
- **How to verify**:
  - Сможете устно описать сценарий «отвалился HTTP‑сервис» и какие шаги сделаете в первую очередь.

---

### Task 9 — Мини‑reverse‑proxy на Nginx (локально)

- **Goal**: Настроить Nginx как reverse proxy к бэкенду, чтобы подготовиться к модулю проекта.
- **Steps**:
  1. Установите Nginx (если ещё не установлен).
  2. Запустите простое приложение на 8080 (тот же `python -m http.server`).
  3. Настройте Nginx так, чтобы:
     - он слушал 80 порт;
     - проксировал запросы на `http://127.0.0.1:8080`.
  4. Проверьте `curl -v http://127.0.0.1/` и проверьте заголовки.
- **Expected result**: Рабочий reverse proxy.
- **How to verify**:
  - `curl` к 80‑му порту возвращает контент с бэкенда;
  - можете показать соответствующий `server { ... proxy_pass ... }` блок.

---

### Task 10 — Диагностика MTU (по желанию)

- **Goal**: Познакомиться с проблемами MTU и фрагментации.
- **Steps**:
  1. Найдите MTU основного интерфейса:

```bash
ip link show
```

  2. Попробуйте:

```bash
ping -c 2 -M do -s 1472 8.8.8.8
```

  3. Уменьшайте `-s`, пока пакеты не начнут проходить.
- **Expected result**: Понимание, что слишком большие пакеты могут дропаться по пути.
- **How to verify**:
  - Можете объяснить, что означают ошибки вида «Frag needed and DF set».

---

## Результаты практики

После выполнения задач вы должны:

- уверенно находить и понимать свои IP‑адреса и маршруты;
- быстро проверять, слушает ли сервис нужный порт и доступен ли он снаружи;
- понимать, как DNS, firewall и локальные настройки (`/etc/hosts`) влияют на доступность;
- иметь в голове чек‑лист из `[[DevOps/11_Networking/troubleshooting]]` на случай «ничего не работает».

Эти навыки напрямую понадобятся вам при работе с Docker‑сетями, Kubernetes‑Service/Ingress и балансировщиками в последующих фазах.
