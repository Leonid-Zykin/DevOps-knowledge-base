## SRE Concepts — SLI/SLO/SLA, error budgets, toil

> ⚡ Tip: SRE = инженеры, управляющие надёжностью через метрики (SLI/SLO) и процессы, а не только «чинящие прод».

### SLI / SLO / SLA

- **SLI (Service Level Indicator)** — измеримая метрика качества (latency, availability, error rate).
- **SLO (Service Level Objective)** — целевое значение SLI (например, успешность 99.9% за 30 дней).
- **SLA (Service Level Agreement)** — контрактный уровень (часто с штрафами).

Примеры SLI:

- доля успешных запросов:
  ```promql
  sum(rate(http_requests_total{code!~"5.."}[5m]))
  /
  sum(rate(http_requests_total[5m]))
  ```
- latency P95 ниже X ms:
  ```promql
  histogram_quantile(
    0.95,
    sum(rate(http_request_duration_seconds_bucket[5m])) by (le)
  )
  ```

### Error Budget

- Если SLO: доступность 99.9% за 30 дней:
  - допустимый простой = 0.1% ≈ 43.2 мин/месяц.
- Ошибка бюджета = фактический простой/ошибки > допустимого.

Практика:

- пока бюджет есть — можно релизить чаще/рискованнее;
- бюджет исчерпан — релизы/изменения замораживаются, фокус на надёжности.

### Toil

- ручная, повторяющаяся, автоматизируемая работа;
- не требует творческого подхода;
- растёт пропорционально размеру системы.

Примеры:

- ручное создание пользователей/окружений;
- повторяющиеся операции деплоя;
- ручной сбор логов/метрик.

Цели:

- уменьшать toil через автоматизацию (CI/CD, self‑service платформы);
- измерять время, тратимое на toil, и ставить цели по его снижению.

## Gotchas

- **Слова «SLO/SLA» без реальных метрик**  
  - **Проблема**: команда и бизнес по-разному понимают «надёжность».  
  - **Решение**: формализовать SLI/SLO, завести дашборды и алерты.

- **Игнорирование error budget'а**  
  - **Проблема**: даже при частых outage продолжаются рискованные релизы.  
  - **Решение**: договориться, что исчерпание бюджета автоматически включает «reliability focus» период.

- **Невидимый toil**  
  - **Проблема**: инженеры выгорают от рутины, но это не видно менеджменту.  
  - **Решение**: трекать toil как отдельную категорию, включать его снижение в OKR/roadmap.

> ⚠️ Warning: Использование SRE‑понятий только как «лейблов» без реального изменения процессов (obs, алерты, error budget policy) создаёт ложное чувство контроля. Надёжность — это про данные и действия, а не только про названия ролей.
