## Архитектурные паттерны — 12‑factor, microservices, blue/green, canary

> ⚡ Tip: DevOps‑инженеру важно понимать архитектуру, чтобы правильно строить CI/CD, мониторинг, сети и SLO.

### 12‑factor app (кратко)

1. Codebase — один репо/источник для сервиса.
2. Dependencies — явные, через менеджеры пакетов.
3. Config — в env‑переменных (см. `[[10_Security/secrets-management]]`).
4. Backing services — как attachable ресурсы (БД, кэш).
5. Build, release, run — разделены.
6. Processes — однотипные, без shared state.
7. Port binding — self‑contained web app.
8. Concurrency — scale‑out через процессы/инстансы.
9. Disposability — быстрый старт/стоп.
10. Dev/prod parity — похожие окружения.
11. Logs — как поток событий (stdout, см. `[[08_Monitoring/loki-and-logging]]`).
12. Admin processes — однократные задачи (Job/CronJob).

### Монолит vs микросервисы

- **Монолит**:
  - проще деплой/отладка;
  - сложнее масштабирование и границы ответственности.
- **Микросервисы**:
  - независимые деплои, команды, стеки;
  - сложная сеть, согласованность, наблюдаемость.

Роль DevOps:

- стандартизированные пайплайны;
- observability‑платформа;
- централизованная инфраструктура (Kubernetes, mesh).

### Blue/Green deployment

- два идентичных окружения: Blue (текущий), Green (новый);
- трафик переключается между ними целиком.

Паттерн:

- деплой в Green → smoke tests → переключение трафика (LB/Ingress/Argo CD) → наблюдение → удалить старое.

### Canary deployment

- пошаговое включение нового релиза для части трафика;
- примеры: 1%, 10%, 50%, 100%.

Инструменты:

- Istio/Linkerd (traffic splitting) (`[[11_Networking/service-mesh]]`);
- Argo Rollouts;
- функционал managed LB (ALB, GCLB, etc).

Связанные: `[[05_CI-CD/concepts]]`, `[[04_Kubernetes/workloads]]`, `[[04_Kubernetes/helm]]`, `[[12_Soft-Skills-and-Processes/sre-concepts]]`.

## Gotchas

- **Микросервисы без платформы и observability**  
  - **Проблема**: хаос, трудно диагностировать и деплоить.  
  - **Решение**: сначала платформа (K8s, CI/CD, мониторинг, логирование), потом активное деление на сервисы.

- **Blue/Green без учёта state**  
  - **Проблема**: разные схемы БД/кэшей между Blue и Green.  
  - **Решение**: миграции «вперёд‑совместимые», feature flags, кросс‑совместимые версии схем.

- **Canary без корректного измерения SLI**  
  - **Проблема**: отклонения не замечаются, canary распространяется на всех.  
  - **Решение**: явные SLI/SLO (`[[12_Soft-Skills-and-Processes/sre-concepts]]`), алерты на canary‑subset, возможность быстрой откатки.

> ⚠️ Warning: Выбор архитектурного паттерна без учёта зрелости команды, процессов и инструментов (например, миграция на микросервисы без платформы и мониторинга) обычно ведёт к росту сложности и падению надёжности. Архитектура и операционные практики должны развиваться синхронно.
