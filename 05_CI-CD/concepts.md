## CI/CD Concepts — пайплайны, артефакты, окружения

> ⚡ Tip: Цель CI/CD — быстрые, повторяемые и безопасные деплои до продакшена, с максимальной автоматизацией проверки качества и безопасности.

### Базовые определения

- **CI (Continuous Integration)**: автоматическая сборка, тестирование и проверка изменений при каждом коммите/PR.
- **CD (Continuous Delivery/Deployment)**:
  - *Delivery*: готовый к деплою, но запуск в прод — по кнопке/ручному подтверждению.
  - *Deployment*: автоматический деплой в прод после прохождения всех проверок.
- **Pipeline**: последовательность стадий (stages/jobs), исполняемых агентами/раннерами.
- **Artifact**: результат сборки (образ, бинарь, архив, chart, manifest).
- **Environment**: логическое окружение (dev, stage, prod), часто привязанное к namespace/кластеру.

### Типичная структура пайплайна

```text
stages:
  - lint
  - test
  - build
  - security
  - deploy
```

Пример шагов:

- **lint**: форматирование, линтеры, статический анализ;
- **test**: unit/integration тесты;
- **build**: сборка бинарей/контейнеров, публикация артефактов;
- **security**: SAST, DAST, dependency/container scanning (см. `[[10_Security/devsecops]]`);
- **deploy**: деплой в окружения (через Helm, Argo CD, Terraform и т.д.).

### Артефакты и кэш

- **artifacts**: результаты job'ов, переиспользуемые в следующих стадиях;
- **cache**: кэш зависимостей (npm/pip/maven), ускоряющий повторные сборки.

Использование:

- GitLab: см. `[[gitlab-ci]]`;
- GitHub Actions: см. `[[github-actions]]`.

### Стратегии деплоя

- **Recreate**: остановить старую версию → поднять новую (простои).
- **Rolling update**: постепенная замена экземпляров без простоя.
- **Blue/Green**: два параллельных окружения, переключение трафика.
- **Canary**: поэтапное включение части трафика на новую версию.

Связанные заметки: `[[12_Soft-Skills-and-Processes/architecture-patterns]]`, `[[04_Kubernetes/workloads]]`, `[[04_Kubernetes/helm]]`, `[[05_CI-CD/argocd]]`.

### Secrets и конфигурация

- Не хранить секреты в открытом виде в CI‑конфигах.
- Использовать:
  - secret‑хранилища (Vault, SOPS, SealedSecrets, cloud secrets) — см. `[[10_Security/secrets-management]]`;
  - механизмы секретов платформы (GitLab CI variables, GitHub Secrets).

### Branch/PR стратегии и CI

- Запускать быстрый CI на:
  - каждом push / merge‑request (lint+unit+build);
- более тяжёлые проверки:
  - nightly, по тегам релизов, по merge в main.

Примеры:

- feature‑ветки → CI + review;
- merge в `main` → build+deploy в stage;
- tag `vX.Y.Z` → build+deploy в prod.

### GitOps

- Истина — в Git (манифесты, Helm charts, Terraform).
- CI пушит изменения в Git‑репо инфраструктуры.
- CD‑система (Argo CD/Flux) синхронизирует состояние кластера с Git.

Связанные: `[[argocd]]`, `[[06_Terraform/state-management]]`, `[[04_Kubernetes/helm]]`.

## Gotchas

- **Слишком тяжёлый CI на каждый коммит**  
  - **Проблема**: пайплайны идут десятки минут, разработка замедляется.  
  - **Решение**: разделять быстрые проверки (на каждый коммит) и тяжёлые (по расписанию/по тегам), оптимизировать кэш.

- **Разный путь деплоя в stage и prod**  
  - **Проблема**: баги проявляются только в проде, т.к. другое «ручное» действие.  
  - **Решение**: сделать деплой максимально одинаковым для всех окружений (разница только в конфиге и gates).

- **Секреты в логах и артефактах CI**  
  - **Проблема**: пароли/ключи попадают в логи пайплайнов, их видят все.  
  - **Решение**: маскирование переменных, отдельные секрет‑хранилища, проверки утечек (`[[10_Security/devsecops]]`).

> ⚠️ Warning: Пайплайны CI/CD имеют доступ к ключам деплоя, реестрам образов, прод‑кластерам и секретам. Любой компрометированный runner или утечка токена может дать атакующему полный доступ к продакшену — относитесь к безопасности CI как к безопасности прод‑инфраструктуры.
