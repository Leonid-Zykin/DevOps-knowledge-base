## Стратегии ветвления: Git Flow, Trunk-based, rebase vs merge

### Базовые понятия

- **trunk/main**: основная стабильная ветка (`main`/`master`).
- **feature‑ветка**: ветка для разработки фичи (`feature/login-ui`).
- **release/hotfix‑ветки**: временные ветки для релизов и фиксов в проде.

> ⚡ Tip: Чем меньше и короче живут ветки, тем проще ревью и мердж, меньше конфликтов.

### Git Flow (классическая модель)

Основные ветки:

- `main` — только прод‑релизы.
- `develop` — основная ветка разработки.

Поддерживающие ветки:

- `feature/*` — новые фичи, ответвляются от `develop`.
- `release/*` — подготовка релиза, от `develop`.
- `hotfix/*` — критичные фиксы, от `main`.

Пример:

```bash
git checkout develop
git checkout -b feature/login
# работа, коммиты
git checkout develop
git merge --no-ff feature/login
git branch -d feature/login
```

Плюсы:

- Чёткая структура, понятные ветки для релизов и хотфиксов.
- Хорошо ложится на «релизы раз в N недель».

Минусы:

- Много долгоживущих веток → больше конфликтов.
- Тяжелее в условиях частых деплоев (CD).

### Trunk-based Development

- Одна основная ветка: `main`/`master` (trunk).
- Короткоживущие feature‑ветки (дни/часы).
- Частые маленькие merge в trunk через PR.
- Неактивный код прячется за feature flags.

Пример:

```bash
git checkout -b feat/login main
# изменения + тесты
git push -u origin feat/login
# PR -> review -> merge (обычно squash merge)
git checkout main
git pull
```

Плюсы:

- Минимум конфликтов, простая модель.
- Отлично подходит для CI/CD и частых релизов.

Минусы:

- Требует дисциплины (тесты, feature flags).
- Не всем командам удобно без «release‑веток».

### rebase vs merge

#### Merge

```bash
git checkout feature/login
git merge main
```

Плюсы:

- Сохраняет реальную историю (кто когда что вливал).
- Безопасно для общих веток.

Минусы:

- История с множеством merge‑коммитов, особенно при долгих ветках.

#### Rebase

```bash
git checkout feature/login
git rebase main
```

Плюсы:

- «Чистая» линейная история.
- Удобно перед merge feature‑ветки в main.

Минусы:

- Переписывает историю — опасно на уже запушенных/используемых ветках.

> ⚠️ Warning: Никогда не делайте `git rebase` общих веток после того, как по ним уже работают другие разработчики.

### Рекомендуемый подход (частый современный вариант)

1. **Trunk-based** с основным `main`:
   - feature‑ветки от `main`;
   - маленькие PR, быстрое ревью и merge.
2. **Merge‑стратегия**:
   - локально: `git fetch && git rebase origin/main` перед push;
   - в GitLab/GitHub: `squash merge` или `rebase+merge`.
3. **Release**:
   - по тегам (`v1.2.3`) и, при необходимости, разовыми `release/*` для крупных релизов.

Сравнение стратегий:

| Стратегия      | Основная ветка | Доп. ветки                     | Когда использовать                     |
|----------------|----------------|--------------------------------|----------------------------------------|
| Git Flow       | main + develop | feature, release, hotfix       | классические релизы раз в N недель    |
| Trunk-based    | main           | короткие feature               | частые деплои, CI/CD                   |
| GitHub Flow    | main           | feature → PR → main            | простые проекты, SaaS                  |

### Feature flags

Вместо долгих веток:

- код фичи мержится в `main` быстро;
- поведение управляется флагом (конфиг, env, фича‑сервис);
- безопасное включение/отключение без деплоя.

Связанные темы: `[[05_CI-CD/concepts]]`, `[[05_CI-CD/github-actions]]`, `[[05_CI-CD/gitlab-ci]]`.

## Gotchas

- **Rebase общих веток**  
  - **Проблема**: коллеги получают конфликты и расхождение истории.  
  - **Решение**: не делать rebase веток, которые уже кем‑то использовались; для них — только merge.

- **Долгоживущие feature‑ветки**  
  - **Проблема**: большие конфликты при merge, сложно ревьюить.  
  - **Решение**: дробить фичи на инкрементальные изменения, мержить часто, использовать feature flags.

- **Смешение Git Flow и Trunk-based**  
  - **Проблема**: часть команды работает через release‑ветки, часть — напрямую в main → хаос.  
  - **Решение**: договориться об одной стратегии, документировать её и ссылаться на эту заметку.

- **Hotfix напрямую в main без backport**  
  - **Проблема**: фикс в проде есть, но не попал в develop/feature‑ветки.  
  - **Решение**: всегда мержить hotfix обратно в основную ветку разработки (develop/main) и помечать тегами.

> ⚠️ Warning: Изменение принятой в команде стратегии ветвления без синхронизации с CI/CD пайплайнами и релизным процессом может привести к «невидимым» коммитам в проде, забытым хотфиксам и разночтениям в артефактах. Обновляйте документацию и пайплайны одновременно.
